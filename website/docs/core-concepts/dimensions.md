# Dimensions

Dimensions represent resource types, properties, or metrics associated with objects and containers. They're the key to expressing multi-resource assignment problems.

## What Are Dimensions?

**Dimensions** are numerical properties that describe:
- **Object dimensions**: Resources required or generated by objects
- **Container dimensions**: Capacity or properties of containers

Think of dimensions as the "currencies" in your assignment problem.

## Why Dimensions Matter

Without dimensions, you can only count objects. With dimensions, you can model:

- **Multi-resource constraints**: "Don't exceed CPU AND memory capacity"
- **Weighted balancing**: "Balance by QPS, not object count"
- **Complex objectives**: "Minimize network latency between objects"
- **Heterogeneous systems**: "Different objects have different sizes"

## Object Dimensions

Object dimensions describe properties of each object.

### Common Examples

| Dimension | Meaning | Domain | Units |
|-----------|---------|--------|-------|
| **Systems Engineering** ||||
| `cpu` | CPU usage | Load balancing | Cores or % |
| `memory` | Memory requirement | Load balancing | GB |
| `qps` | Queries per second | Database | Requests/sec |
| **Operations & Logistics** ||||
| `weight` | Package weight | Delivery | kg or lbs |
| `volume` | Package volume | Warehouse | cubic meters |
| `distance` | Route distance | Transportation | miles or km |
| `time_required` | Task duration | Scheduling | minutes |
| **Healthcare** ||||
| `care_level` | Acuity level | Patient assignment | 1-5 scale |
| `isolation_required` | Needs isolation | Patient assignment | 0 or 1 |
| `procedure_duration` | Surgery length | OR scheduling | minutes |
| **Education** ||||
| `student_preference` | Course preference | Class assignment | 1-5 ranking |
| `skill_level` | Student proficiency | Course placement | Beginner-Advanced |
| `credit_hours` | Course workload | Scheduling | Hours |

## Container Dimensions

Container dimensions describe properties or capacities of containers.

### Common Examples

| Dimension | Meaning | Domain | Units |
|-----------|---------|--------|-------|
| **Systems Engineering** ||||
| `cpu_capacity` | CPU available | Load balancing | Cores |
| `memory_capacity` | Memory available | Load balancing | GB |
| `network_capacity` | Network bandwidth | CDN | Gbps |
| **Operations & Logistics** ||||
| `vehicle_capacity` | Cargo capacity | Delivery | kg or cubic meters |
| `max_deliveries` | Max stops per route | Routing | Count |
| `operating_hours` | Available time | Scheduling | Hours per day |
| **Healthcare** ||||
| `bed_capacity` | Number of beds | Hospital ward | Count |
| `staff_count` | Nurses available | Shift assignment | Count |
| `equipment_available` | Has required equipment | Procedure scheduling | 0 or 1 |
| **Education** ||||
| `class_size_limit` | Max students | Course section | Count |
| `teacher_expertise` | Subject expertise | Teacher assignment | Skill level |
| `room_capacity` | Seats available | Classroom | Count |

## Automatic Dimension Aggregation

When you define object dimensions, Rebalancer automatically aggregates them at the container level.

### Example

```python
# Define per-object CPU usage
solver.add_object_dimension("cpu", {
    "task0": 2.0,
    "task1": 1.5,
})

# If task0 and task1 are assigned to host0,
# Rebalancer automatically computes:
#   host0's "cpu" utilization = 2.0 + 1.5 = 3.5
```

This aggregated dimension can be used in constraints and goals:

```python
from rebalancer.specs import CapacitySpec, ConstraintSpec

# Ensure CPU usage doesn't exceed capacity
solver.add_constraint(
    ConstraintSpec(
        capacitySpec=CapacitySpec(
            name="cpu-cap",
            scope="host",
            dimension="cpu",  # Uses aggregated object cpu
        )
    )
)
```

## Multi-Dimensional Constraints

You can constrain multiple dimensions simultaneously.

### Example: CPU and Memory

```python
from rebalancer.specs import (
    BalanceSpec,
    CapacitySpec,
    ConstraintSpec,
    GoalSpec,
)

# Define both dimensions
solver.add_object_dimension("cpu", cpu_per_task)
solver.add_object_dimension("memory", memory_per_task)

solver.add_container_dimension("cpu_capacity", cpu_per_host)
solver.add_container_dimension("memory_capacity", memory_per_host)

# Constraint 1: Respect CPU capacity
solver.add_constraint(
    ConstraintSpec(
        capacitySpec=CapacitySpec(
            name="cpu_capacity", scope="host", dimension="cpu"
        )
    )
)

# Constraint 2: Respect memory capacity
solver.add_constraint(
    ConstraintSpec(
        capacitySpec=CapacitySpec(
            name="memory_capacity", scope="host", dimension="memory"
        )
    )
)

# Goal: Balance CPU usage
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(
            name="balance_cpu", scope="host", dimension="cpu"
        )
    ),
    weight=1.0,
)
```

Both constraints must be satisfied. The goal optimizes CPU balance subject to both capacity constraints.

## Dimension Naming Conventions

### Best Practices

1. **Be descriptive**: `memory_gb` not `mem`
2. **Include units in name**: `network_mbps`, `latency_ms`
3. **Use consistent naming**: `cpu`, `memory`, `disk` (not mixing `cpu_usage` and `mem`)
4. **Lowercase with underscores**: `network_bandwidth`, not `NetworkBandwidth`

### Matching Object and Container Dimensions

For capacity constraints, object and container dimensions should align:

- Object dimension: `cpu` (usage)
- Container dimension: `cpu_capacity` (limit)
- Or use same name: `cpu` for both (usage and capacity)

Rebalancer doesn't enforce naming - you specify which dimensions to compare in each spec.

## Special Dimension Types

### Built-In: `{objectName}_count`

Rebalancer automatically provides an implicit count dimension based on the object name you set with `setObjectName()`. Each object has a value of 1.0 for this dimension.

**The dimension name is**: `{objectName}_count`

For example:
- If you call `setObjectName("task")`, the dimension is `task_count`
- If you call `setObjectName("shard")`, the dimension is `shard_count`
- If you call `setObjectName("object")`, the dimension is `object_count`

```python
from rebalancer.specs import BalanceSpec, GoalSpec

# Example 1: Using task_count
solver.set_object_name("task")
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(
            name="balance_count",
            scope="host",
            dimension="task_count",  # Matches set_object_name("task")
        )
    ),
    weight=1.0,
)

# Example 2: Using shard_count
solver.set_object_name("shard")
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(
            name="balance_count",
            scope="server",
            dimension="shard_count",  # Matches set_object_name("shard")
        )
    ),
    weight=1.0,
)
```

**You don't need to define this dimension** - it's created automatically with a value of 1.0 for every object.

## Dimension Units and Scaling

### Units Don't Matter (Mathematically)

Rebalancer doesn't care about units - it just works with numbers. But **you** should care:

1. **Consistency**: Mixing units makes debugging harder
2. **Numerical stability**: MIP solvers (like the Optimal Solver) can suffer from numerical instability when working with very large numbers or very large ranges. Use reasonably-scaled values (e.g., GB instead of bytes, hours instead of milliseconds) to avoid precision issues.

```python
# BAD: Mixing units
cpu_cores = {"task0": 2.0}        # In cores
memory_gb = {"task1": 2000.0}     # In MB (should be GB!)

# ALSO BAD: Very large numbers can cause MIP solver instability
memory_bytes = {"task0": 4000000000.0}  # 4GB in bytes - too large!
latency_ns = {"task0": 50000000.0}      # 50ms in nanoseconds - too large!
```

### Keep Units Consistent

```python
# GOOD: Consistent units
cpu_cores = {"task0": 2.0}        # In cores
memory_gb = {"task0": 4.0}        # In GB
network_gbps = {"task0": 0.5}     # In Gbps
```

### Scaling for Goal Weights

When balancing multiple dimensions, consider their scales:

```python
# CPU: ranges from 1-8 cores
# Memory: ranges from 1-64 GB
# If you weight them equally, memory dominates (bigger numbers)

# Solution: Balance both but weight CPU higher
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(name="cpu", scope="host", dimension="cpu")
    ),
    weight=8.0,
)
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(name="mem", scope="host", dimension="memory")
    ),
    weight=1.0,
)

# Or normalize dimensions to 0-1 range before adding them
```

## Utilization vs. Capacity

For capacity constraints, you typically need both:

| Concept | Type | Example Dimension |
|---------|------|-------------------|
| **Utilization** | Object dimension (aggregated) | `memory` |
| **Capacity** | Container dimension | `memory_capacity` |

### Example

```python
from rebalancer.specs import CapacitySpec, ConstraintSpec

# Object dimension: how much each object uses
solver.add_object_dimension("memory", {
    "task0": 2.0,  # Uses 2GB
    "task1": 4.0,  # Uses 4GB
})

# Container dimension: how much each container has
solver.add_container_dimension("memory_capacity", {
    "host0": 16.0,  # Has 16GB
    "host1": 32.0,  # Has 32GB
})

# Constraint: utilization <= capacity
solver.add_constraint(
    ConstraintSpec(
        capacitySpec=CapacitySpec(
            name="memory_limit",
            scope="host",
            dimension="memory",  # Utilization (object dimension)
            # Automatically compared against "memory_capacity" container dimension
        )
    )
)
```

**Automatic Capacity Matching**: CapacitySpec automatically looks for a matching capacity dimension by trying:
1. `{dimension}_capacity` (e.g., `memory` → `memory_capacity`)
2. The dimension name itself as a container dimension

So if your utilization dimension is `memory`, Rebalancer will look for a container dimension called `memory_capacity` first, then fall back to `memory` if not found.

See [CapacitySpec reference](../reference/capacity) for details on relative limits and other capacity options.

## Common Patterns

### Pattern 1: Homogeneous Objects

All objects identical:

```python
# All tasks use 1.0 CPU and 2.0 GB
solver.add_object_dimension("cpu", {}, default_value=1.0)
solver.add_object_dimension("memory", {}, default_value=2.0)
```

### Pattern 2: Binary Flags

Use 0/1 for boolean properties:

```python
from rebalancer.specs import CapacitySpec, ConstraintSpec, Limit

# Is this task latency-sensitive?
solver.add_object_dimension(
    "latency_sensitive",
    {
        "task_api": 1.0,      # Yes
        "task_batch": 0.0,    # No
    },
    default_value=0.0,
)

# Constraint: Limit latency-sensitive tasks per host
solver.add_constraint(
    ConstraintSpec(
        capacitySpec=CapacitySpec(
            name="limit_latency_sensitive",
            scope="host",
            dimension="latency_sensitive",
            limit=Limit(type="ABSOLUTE", globalLimit=5.0),  # Max 5 per host
        )
    )
)
```

### Pattern 3: Relative Sizes

Use relative weights when absolute values don't matter:

```python
from rebalancer.specs import BalanceSpec, GoalSpec

# Task sizes: small=1, medium=2, large=4
solver.add_object_dimension("size", {
    "task_small": 1.0,
    "task_medium": 2.0,
    "task_large": 4.0,
})

# Balance by total size (not count)
solver.add_goal(
    GoalSpec(
        balanceSpec=BalanceSpec(name="balance-size", scope="host", dimension="size")
    ),
    weight=1.0,
)
```

### Pattern 4: Cost Optimization

Dimensions don't have to be resources - can be costs:

```python
from rebalancer.specs import GoalSpec, MaximizeAllocationSpec

# Cost of running each task on each host type
solver.add_dynamic_object_dimension(
    "cost",
    scope="host_type",
    scope_item_to_object_to_value={
        "spot": {"task0": 1.0, "task1": 1.0},        # Cheap
        "on_demand": {"task0": 3.0, "task1": 3.0},   # Expensive
    },
)

# Minimize total cost: use MaximizeAllocationSpec with a negative weight,
# since there is no MinimizeAllocationSpec in the API.
solver.add_goal(
    GoalSpec(
        maximizeAllocationSpec=MaximizeAllocationSpec(
            name="minimize-cost", scope="host_type", dimension="cost"
        )
    ),
    weight=-1.0,
)
```
